# Gallery Auto-Sync Design

**Date:** 2026-06-30  
**Status:** Approved

## Problem

Adding images to a gallery requires two manual steps: copying the image into `images/galleries/<name>/` and then editing `_galleries/<name>.yml` to add the new entry. This is error-prone and easy to forget, leading to images sitting in the folder with no YML entry (and therefore not appearing on the site).

## Solution

A GitHub Action runs automatically whenever images are pushed to any folder under `images/galleries/`. A Python script diffs the folder contents against the YML, renames new images with an order prefix, and appends entries for any images not yet listed. An optional per-gallery metadata file can supply richer titles and descriptions — written by hand or generated by an LLM — which the script uses in place of defaults.

## Architecture

```
Push to main (images/galleries/**)
        │
        ▼
GitHub Action triggered (.github/workflows/sync-gallery.yml)
        │
        ▼
Python script: .github/scripts/sync_gallery.py
  1. Scan all subdirectories under images/galleries/
  2. For each folder, load matching _galleries/<name>.yml
  3. Load _galleries/<name>.meta.yml if it exists (optional)
  4. Diff: find images in folder not present in YML
  5. Sort new images alphabetically
  6. For each new image:
     a. Determine next order number
     b. Rename file to <order>_<original> (e.g. 14_IMG_9780.png)
     c. Use meta values where present, defaults otherwise
     d. Append new YML entry referencing renamed file
  7. Write updated YML
        │
        ▼
Action creates branch auto/gallery-sync-<YYYYMMDD-HHMMSS>
commits renamed files + updated YML
opens PR to main: "auto: sync gallery"
(entire step skipped if no new images detected)
        │
        ▼
Aaron reviews PR, merges when satisfied
```

## Files

| File | Purpose |
|---|---|
| `.github/workflows/sync-gallery.yml` | GitHub Action definition |
| `.github/scripts/sync_gallery.py` | Sync logic |
| `_galleries/<name>.meta.yml` | Optional per-gallery metadata (LLM-generated or hand-written) |

## Script Behaviour

**Scope:** Loops over every folder under `images/galleries/` and finds its matching `_galleries/<name>.yml`. Fully generic — new galleries are picked up automatically with no script changes.

**Supported image extensions:** `.jpg`, `.jpeg`, `.png`, `.gif`, `.webp`

**New entry defaults:**

| Field | Value |
|---|---|
| `image` | `images/galleries/<name>/<order>_<filename>` |
| `title` | Original filename without extension (e.g. `IMG_9780.png` → `IMG_9780`) |
| `artist` | `Aaron Liu` |
| `description` | `tbc` |
| `order` | Max existing `order` value + 1, incrementing per new file |

**File renaming:** New images are renamed with a zero-padded 2-digit order prefix before being committed (e.g. `IMG_9780.png` → `14_IMG_9780.png`). The `order` value in the YML matches the prefix number. Existing files already present in the YML are never renamed.

**Ordering:** New images are sorted alphabetically before being processed, so order assignment is deterministic across runs.

**Safety:** Existing YML entries are never modified or removed. The script is append-only.

## Optional Metadata File

Each gallery may have a `_galleries/<name>.meta.yml` file keyed by the **original filename** (before renaming). Any field present overrides the corresponding default; missing fields fall back to defaults.

**Example `_galleries/illustration.meta.yml`:**
```yaml
IMG_9776.png:
  title: "Claude Monet Master Copy"
  description: "Acrylic on paper, inspired by Monet's water lilies series"

IMG_9777.png:
  title: "Manet Master Copy"
  description: "Oil on paper study"
  artist: "Guest Artist"
```

**Meta file rules:**
- Keyed by original filename so users and LLMs can write metadata against the name they see when uploading, before renaming occurs.
- Append-only: when an LLM (or any tool) regenerates the meta file, it only adds entries for images not already present. Existing entries — including any manual corrections — are preserved.
- The Action never modifies the meta file. It is a read-only input to the sync script.
- The meta file is committed to the repo alongside the images.

## GitHub Action

```yaml
on:
  push:
    branches: [main]
    paths:
      - 'images/galleries/**'
```

Steps:
1. Checkout repo with write permissions (`contents: write`, `pull-requests: write`)
2. Run `sync_gallery.py`
3. If any files changed:
   - Create branch `auto/gallery-sync-<YYYYMMDD-HHMMSS>`
   - Commit renamed images + updated YML with message `auto: sync gallery`
   - Push branch
   - Open PR to main using `gh pr create` with a summary of galleries and images added
4. If no files changed, exit silently — no branch, no PR

Each Action run produces at most one PR. Aaron reviews the diff, adjusts titles/descriptions in the YML if needed, and merges.

## Out of Scope

- Removing YML entries when images are deleted (manual cleanup)
- Reordering existing entries
- Renaming files that are already in the YML
- Running the LLM itself — the Action only reads the meta file, not generates it
- Syncing galleries other than those with a matching `_galleries/<name>.yml`
